## Что такое CosmosDb ? 

`Cosmos DB` - это облачная база данных от Microsoft, которая поддерживает различные модели данных, включая документы, графы и ключ-значение. Она предназначена для высокопроизводительных и масштабируемых приложений, обеспечивая низкую задержку и высокую доступность. 

Ее отличие от того же самого `MS SQL` заключается в том, что `Cosmos DB` - это нереляционная база данных, которая позволяет хранить данные в формате JSON и поддерживает различные API для работы с данными, такие как SQL, MongoDB, Cassandra и другие.

## Отличия между реляционными и нереляционными БД

Реляционные базы данных (например, MS SQL) используют таблицы для хранения данных, где каждая таблица имеет фиксированную структуру и связи между таблицами. Это позволяет легко выполнять сложные запросы и обеспечивать целостность данных.
Нереляционные базы данных (например, Cosmos DB) используют более гибкие структуры данных, такие как документы, графы или ключ-значение. Это позволяет легко масштабировать и изменять структуру данных, но может усложнить выполнение сложных запросов и обеспечить целостность данных.

В таких базах данных считывание происходит быстрее и куда проще хранить например текстовую информацию. 

Запомните, при работе с проектом в независимости от его типа(монолит или микросервисы) вы можете для разных задач использовать разные базы данных. Например, для хранения текстовой информации вы можете использовать `Cosmos DB`, а для хранения структурированных данных - `MS SQL`. Как пример могу предложить вам новостной сайт, где вы можете использовать `Cosmos DB` для хранения текстов новостей, а `MS SQL` для хранения информации о пользователях и их действиях на сайте.


## Структура NoSQL CosmosDb

В такой структуре у каждого документа есть уникальный идентификатор, который позволяет быстро находить и извлекать данные. Документы могут содержать вложенные объекты и массивы, что позволяет хранить сложные структуры данных. 

Структура выглядит следующим образом:
База данных => Контейнер => Документ 

У каждого документа есть уникальный идентификатор, который позволяет быстро находить и извлекать данные. Документы могут содержать вложенные объекты и массивы, что позволяет хранить сложные структуры данных. Также у него есть `partition key`, который позволяет разбивать данные на части и хранить их в разных местах, что позволяет быстро находить и извлекать данные. 

```json
{
    "id": "aaaaaaaa-0000-1111-2222-bbbbbbbbbbbb",
    "category": "gear-surf-surfboards",
    "name": "Yamba Surfboard",
    "quantity": 12,
    "price": 850,
    "clearance": false,
    "_rid": "kCUWAKW1JTQBAAAAAAAAAA==",
    "_self": "dbs/kCUWAA==/colls/kCUWAKW1JTQ=/docs/kCUWAKW1JTQBAAAAAAAAAA==/",
    "_etag": "\"0200df7b-0000-0a00-0000-6814afcc0000\"",
    "_attachments": "attachments/",
    "_ts": 1746186188
}
```

К примеру здесь у нас `category` - это `partition key`, а `id` - это уникальный идентификатор документа. 

Мы сделали `category` `partition key`, чтобы мы могли находить нужную информацию по ней. Например, если нам нужно найти все товары в категории `gear-surf-surfboards`, мы можем использовать `category` как `partition key`, чтобы быстро найти все документы с этой категорией. Можете считать это аналогом `index` в реляционных базах данных. 


## RequestCharge 

`RequestCharge` - это метрика, которая показывает, сколько ресурсов было использовано для выполнения запроса к базе данных. Она измеряется в `RU/s` (Request Units per second) и позволяет оценить стоимость выполнения запроса.
