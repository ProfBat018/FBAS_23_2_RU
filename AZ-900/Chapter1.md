# AZ-900: Chapter 1.1 - Identify the benefits and considerations of using cloud services

## Benefits of using cloud services

### High Availability
- Гарантия доступности приложений и сервисов 24/7 даже при сбоях оборудования.
- **Пример:** Azure предоставляет Service Level Agreements (SLAs) с гарантией 99.99% времени безотказной работы.

### Scalability
- Возможность увеличивать или уменьшать ресурсы по мере необходимости.
- **Vertical Scaling (Scale Up):** Увеличение мощности одной машины (например, добавление CPU или RAM).
- **Horizontal Scaling (Scale Out):** Добавление дополнительных машин для распределения нагрузки.

### Elasticity
- Автоматическое масштабирование ресурсов вверх или вниз в зависимости от текущего спроса.
- **Пример:** Azure App Service может автоматически масштабироваться при росте количества пользователей.

### Agility
- Способность быстро разрабатывать, тестировать и внедрять новые решения без долгой подготовки инфраструктуры.
- **Пример:** Быстрая развертка серверов через Azure Marketplace.

### Fault Tolerance
- Возможность системы продолжать работу при отказе отдельных её компонентов.
- **Пример:** Использование Availability Zones для обеспечения отказоустойчивости приложений.

### Disaster Recovery
- Возможность быстрого восстановления данных и сервисов после катастрофических сбоев.
- **Пример:** Azure Site Recovery автоматизирует процесс восстановления серверов и баз данных.

### Capital Expenditure (CapEx) vs Operational Expenditure (OpEx)
- **CapEx:** Большие единовременные затраты на покупку оборудования.
- **OpEx:** Оплата только за фактическое использование облачных ресурсов без крупных начальных вложений.

### Economies of Scale
- Снижение стоимости за счёт использования масштабируемой инфраструктуры, обслуживающей множество клиентов.
- **Пример:** Microsoft может снижать стоимость сервисов благодаря огромной сети дата-центров.

---

## Considerations when using cloud services

### Internet Dependency
- Для работы облачных сервисов требуется стабильное интернет-соединение.

### Less Control
- Ограниченный доступ к физической инфраструктуре и некоторым уровням настройки.

### Security Responsibilities
- Ответственность за безопасность данных разделяется между облачным провайдером и клиентом (Shared Responsibility Model).

### Regulatory Compliance
- Необходимость соблюдения требований законодательства в области хранения и обработки данных (например, GDPR).

### Cost Management
- При неправильном управлении использование ресурсов может привести к неожиданно высоким расходам.

---

## Summary

> Использование облачных сервисов даёт высокую доступность, гибкость, экономичность и масштабируемость, но требует внимательного подхода к вопросам безопасности, соблюдения нормативных требований и управления затратами.


# AZ-900: Chapter 1.2 - Describe the differences between categories of cloud services

## Infrastructure as a Service (IaaS)

**Что это:**
- Модель облачных сервисов, где клиент арендует базовую инфраструктуру: виртуальные машины, сети, хранилища данных.
- Провайдер управляет только "железом", а клиент сам устанавливает операционные системы, middleware, приложения и данные.

**Преимущества:**
- Полный контроль над операционной системой и приложениями.
- Гибкость в настройке окружения.
- Быстрая замена оборудования без капитальных затрат.

**Примеры:**
- Azure Virtual Machines
- Azure Virtual Network
- Azure Load Balancer

---

## Platform as a Service (PaaS)

**Что это:**
- Модель, где клиенту предоставляется готовая платформа для разработки, тестирования и развертывания приложений.
- Провайдер управляет всей инфраструктурой, операционной системой, базами данных и серверной средой.

**Преимущества:**
- Меньше задач по управлению инфраструктурой.
- Ускорение процесса разработки.
- Интеграция с DevOps процессами и CI/CD.

**Примеры:**
- Azure App Service
- Azure SQL Database
- Azure Functions

---

## Software as a Service (SaaS)

**Что это:**
- Модель, где конечные пользователи получают готовое приложение через интернет без забот о серверах, обновлениях и поддержке.
- Всё обслуживание, обновления и безопасность берет на себя провайдер.

**Преимущества:**
- Мгновенный доступ к приложениям.
- Нет необходимости в установке и обслуживании.
- Скалируемость по количеству пользователей.

**Примеры:**
- Microsoft 365 (Office 365)
- Azure DevOps Services
- Outlook.com

---

## Сводная таблица сравнения

| Модель | Кто управляет инфраструктурой | Кто управляет приложением |
|:-|:-|:-|
| IaaS | Провайдер | Клиент |
| PaaS | Провайдер | Частично клиент (код) |
| SaaS | Провайдер | Провайдер |

---

## Краткий вывод

> IaaS дает максимум контроля, PaaS ускоряет разработку, а SaaS полностью освобождает пользователя от забот о поддержке и инфраструктуре.

# AZ-900: Chapter 1.3 - Describe the differences between types of cloud computing

## Public Cloud

**Что это:**
- Модель облачных вычислений, где ресурсы (серверы, хранилища, сети) принадлежат и управляются сторонним облачным провайдером.
- Пользователи арендуют и используют эти ресурсы через интернет.

**Преимущества:**
- Нет затрат на покупку или управление оборудованием.
- Высокая масштабируемость.
- Быстрая доступность ресурсов.

**Недостатки:**
- Ограниченный контроль над инфраструктурой.
- Возможные требования к безопасности и соответствию законодательству.

**Примеры:**
- Microsoft Azure
- Amazon Web Services (AWS)
- Google Cloud Platform (GCP)

---

## Private Cloud

**Что это:**
- Облачная инфраструктура используется только одной организацией.
- Может находиться в собственном дата-центре компании или у внешнего провайдера.

**Преимущества:**
- Повышенный уровень контроля и настройки.
- Более строгая безопасность и соответствие требованиям отрасли.

**Недостатки:**
- Высокая стоимость создания и обслуживания.
- Требуются собственные ресурсы для управления инфраструктурой.

**Примеры:**
- Частный облак на базе Azure Stack
- Собственная виртуализированная инфраструктура компании

---

## Hybrid Cloud

**Что это:**
- Комбинация публичного и приватного облаков с возможностью передачи данных и приложений между ними.
- Позволяет организациям сохранять часть ресурсов в частном облаке и использовать публичное облако для масштабируемости.

**Преимущества:**
- Гибкость в размещении рабочих нагрузок.
- Оптимизация затрат.
- Улучшенная поддержка требований к безопасности и соответствию.

**Недостатки:**
- Сложность интеграции и управления.
- Требует тщательной настройки безопасности и сетевых связей.

**Примеры:**
- Использование Azure Arc для управления как публичными, так и приватными ресурсами.
- Хранение чувствительных данных локально, а общих данных — в публичном облаке.

---

## Сводная таблица сравнения

| Модель | Характеристики | Примеры |
|:-|:-|:-|
| Public Cloud | Ресурсы делятся между организациями, управляются провайдером | Azure, AWS, GCP |
| Private Cloud | Используется одной организацией, высокий контроль | Azure Stack, частные ЦОД |
| Hybrid Cloud | Комбинация публичного и приватного облаков | Azure Arc, гибридные сценарии |

---

## Краткий вывод

> Public Cloud даёт доступность и масштабируемость, Private Cloud обеспечивает контроль и безопасность, а Hybrid Cloud сочетает в себе лучшее от обеих моделей, но требует сложной интеграции.
