## Docker and OCI

С прошлых уроков мы уже знаем что такое `контейнеры` и как они работают. Но как же они реализованы? Для этого нам нужно познакомиться с Docker и OCI.

`OCI` - Open Container Initiative

OCI - это некоммерческая организация, которая разрабатывает открытые стандарты для контейнеров. Она была создана в 2015 году и включает в себя таких участников как Docker, Google, Microsoft, Red Hat, IBM и многих других. OCI разрабатывает два стандарта: Runtime Specification и Image Specification.

`Runtime Specification` - это спецификация, которая описывает как должен работать контейнер. Она описывает как должен работать контейнер, какие должны быть у него namespace, какие cgroups и т.д.

`Image Specification` - это спецификация, которая описывает как должен выглядеть образ контейнера. Она описывает как должен выглядеть образ, какие должны быть у него слои, какие должны быть у него метаданные и т.д.

## Namespaces

`Namespaces` - это механизм ядра Linux, который позволяет изолировать процессы друг от друга. В Linux есть 7 типов namespaces:

- `PID` - Process изолирует процессы друг от друга. Каждый процесс внутри контейнера имеет свой уникальный PID, который не пересекается с PID процессов вне контейнера.
- `Network` - изолирует сетевые интерфейсы, таблицы маршрутизации, правила файрвола и т.д. Каждый контейнер имеет свой уникальный набор сетевых интерфейсов, таблиц маршрутизации и т.д. В будущем мы поговорим подробнее о сетевых интерфейсах, например про IpvNet & MacVnet
- `Mount` - изолирует точки монтирования файловой системы. Каждый контейнер имеет свой уникальный набор точек монтирования.
- `UTS` - изолирует имена хоста и доменное имя. Каждый контейнер имеет свое уникальное имя хоста и доменное имя.
- `IPC` - изолирует межпроцессное взаимодействие. Каждый контейнер имеет свой уникальный набор IPC ресурсов. То есть по факту наши контейнеры не могут общаться друг с другом, потому что они не имеют общих IPC ресурсов.
- `User` - изолирует пользователей и группы. Каждый контейнер имеет свой уникальный набор пользователей и групп.
- `Cgroup` - изолирует ресурсы. Каждый контейнер имеет свой уникальный набор ресурсов.

Если провести параллель с виртуальными машинами, то можно сказать что namespaces - это то же самое что и виртуализация аппаратных ресурсов. То есть виртуальные машины изолируют аппаратные ресурсы, а контейнеры изолируют процессы.

Если провести параллель с программированием, то там мы тоже изолируем классы, структуры, переменные и т.д. То есть в программировании мы изолируем данные, а в контейнерах мы изолируем процессы.

## UnionFS

`UnionFS` - это механизм ядра Linux, который позволяет объединять несколько файловых систем в одну. В Linux есть несколько типов UnionFS:

- `AUFS` - Advanced Multi-Layered Unification Filesystem
- `OverlayFS` - Overlay Filesystem
- `DeviceMapper` - Device Mapper
- `Btrfs` - B-tree File System
- `ZFS` - Z File System

## Docker Engine

Сама система Docker Engine состоит из нескольких компонентов:

- `Docker Daemon` - это демон, который запускается на хосте и управляет контейнерами. Он отвечает за создание, запуск, остановку и удаление контейнеров. Также он отвечает за создание, загрузку и удаление образов. Демон слушает сокеты и REST API.
- `Docker Client` - это клиент, который запускается на хосте и управляет демоном. Он отвечает за создание, запуск, остановку и удаление контейнеров. Также он отвечает за создание, загрузку и удаление образов. Клиент подключается к демону через сокеты или REST API.
- `Docker Registry` - это реестр, который хранит образы. Он отвечает за загрузку и удаление образов. Также он отвечает за авторизацию и аутентификацию пользователей. Реестр может быть публичным или приватным. Публичный реестр доступен всем, а приватный реестр доступен только авторизованным пользователям.

и все это работает на `Linux`.

## Dockerfile

```Dockerfile

FROM mcr.microsoft.com/dotnet/aspnet:9.0 AS base
USER $APP_UID
WORKDIR /app
EXPOSE 80

FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
ARG BUILD_CONFIGURATION=Release
WORKDIR /src
COPY ["Intro/Intro.csproj", "Intro/"]
RUN dotnet restore "Intro/Intro.csproj"
COPY . .
WORKDIR "/src/Intro"
RUN dotnet build "./Intro.csproj" -c $BUILD_CONFIGURATION -o /app/build

FROM build AS publish
ARG BUILD_CONFIGURATION=Release
RUN dotnet publish "./Intro.csproj" -c $BUILD_CONFIGURATION -o /app/publish /p:UseAppHost=false

FROM base AS final
WORKDIR /app
COPY --from=publish /app/publish .
ENTRYPOINT ["dotnet", "Intro.dll"]


```

DockerFile - это файл, который описывает как должен выглядеть ваш образ. Он состоит из нескольких команд, которые выполняются последовательно. Каждая команда создает новый слой(layer) в образе. Когда вы создаете образ, Docker выполняет команды в DockerFile и создает новый образ. Когда вы запускаете контейнер, Docker создает новый контейнер на основе образа.

**Что такое слой ?**

Слой - это файл, который содержит изменения, которые были внесены в образ. Каждый слой содержит только те изменения, которые были внесены в образ. Также учитывая то, что ваш докерфайл запускается последовательно, то каждый слой будет зависеть от предыдущего. То есть если вы измените один слой, то все слои которые зависят от него тоже будут изменены. Поэтому если вы хотите изменить один слой, то вам нужно будет пересоздать все слои которые зависят от него. Это называется `кэширование`(caching). Docker кэширует слои и использует их повторно, если они не были изменены. Это позволяет ускорить процесс создания образов.

Давайте разберем каждый слой в нашем DockerFile: 

```Dockerfile

FROM mcr.microsoft.com/dotnet/aspnet:9.0 AS base
USER $APP_UID
WORKDIR /app
EXPOSE 80
```

- `FROM` - это команда, которая указывает от какого образа мы будем создавать наш образ. В данном случае мы используем образ `mcr.microsoft.com/dotnet/aspnet:9.0`, который содержит ASP.NET Core 9.0.
- `USER` - это команда, которая указывает от какого пользователя мы будем запускать наш образ. В данном случае мы используем переменную окружения `$APP_UID`, которая содержит UID пользователя. **Эту строку надо удалть так как мы не создаеи никакого юзера**
- `WORKDIR` - это команда, которая указывает рабочую директорию для нашего образа. В данном случае мы используем директорию `/app`. По сути всегда когда мы будем запускать контейнер, он будет запускаться из этой директории. 
- `EXPOSE` - это команда, которая указывает какие порты мы будем открывать в нашем образе. В данном случае мы открываем порт 80. Это значит что когда мы будем запускать контейнер, он будет слушать на порту 80. Вы можете открыть сразу несколько портов. 


```Dockerfile
FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
ARG BUILD_CONFIGURATION=Release
WORKDIR /src
COPY ["Intro/Intro.csproj", "Intro/"]
RUN dotnet restore "Intro/Intro.csproj"
COPY . .
WORKDIR "/src/Intro"
RUN dotnet build "./Intro.csproj" -c $BUILD_CONFIGURATION -o /app/build
```

- `FROM` - это команда, которая указывает от какого образа мы будем создавать наш образ. В данном случае мы используем образ `mcr.microsoft.com/dotnet/sdk:9.0`, который содержит .NET SDK 9.0.
- `ARG` - это команда, которая указывает переменную окружения для нашего образа. В данном случае мы используем переменную окружения `BUILD_CONFIGURATION`, которая содержит конфигурацию сборки. По умолчанию она равна `Release`.
- `WORKDIR` - это команда, которая указывает рабочую директорию для нашего образа. В данном случае мы используем директорию `/src`. По сути всегда когда мы будем запускать контейнер, он будет запускаться из этой директории.
- `COPY` - это команда, которая копирует файлы из локальной директории в образ. В данном случае мы копируем файл `Intro.csproj` из локальной директории в директорию `Intro/` в образе.
- `RUN` - это команда, которая выполняет команду в образе. В данном случае мы выполняем команду `dotnet restore`, которая восстанавливает зависимости для проекта `Intro.csproj`.
- `COPY` - это команда, которая копирует файлы из локальной директории в образ. В данном случае мы копируем все файлы из локальной директории в директорию `/src/` в образе.
- `WORKDIR` - это команда, которая указывает рабочую директорию для нашего образа. В данном случае мы используем директорию `/src/Intro`. По сути всегда когда мы будем запускать контейнер, он будет запускаться из этой директории.
- `RUN` - это команда, которая выполняет команду в образе. В данном случае мы выполняем команду `dotnet build`, которая собирает проект `Intro.csproj` и сохраняет его в директорию `/app/build`.


```Dockerfile
FROM build AS publish
ARG BUILD_CONFIGURATION=Release
RUN dotnet publish "./Intro.csproj" -c $BUILD_CONFIGURATION -o /app/publish /p:UseAppHost=false
```
- `FROM` - это команда, которая указывает от какого образа мы будем создавать наш образ. В данном случае мы используем образ `build`, который был создан на предыдущем шаге.
- `ARG` - это команда, которая указывает переменную окружения для нашего образа. В данном случае мы используем переменную окружения `BUILD_CONFIGURATION`, которая содержит конфигурацию сборки. По умолчанию она равна `Release`.
- `RUN` - это команда, которая выполняет команду в образе. В данном случае мы выполняем команду `dotnet publish`, которая публикует проект `Intro.csproj` и сохраняет его в директорию `/app/publish`. Также мы указываем что нам не нужен AppHost, так как мы будем запускать контейнер на Linux.


Есть несколько способов `build`-а образа: 
- Через `GUI` в `Rider` или же `VS`
- Через `CLI` в терминале 


```bash
docker build -t intro:latest .
```
- `docker build` - это команда, которая создает образ из DockerFile.
- `-t` - это флаг, который указывает тег для образа. В данном случае мы используем тег `intro:latest`, который указывает что это последний образ.
- `.` - это путь к DockerFile. В данном случае мы используем текущую директорию.
если вы не хотите указывать тег, то вы можете указать `-t` без тега и тогда Docker сам проставит тег `latest`.
