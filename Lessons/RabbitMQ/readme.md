# Message brokers 

С микросервисной архитектурой мы уже разобрались и тут нужно понять несколько важных вещей. Если сервисы общаются между собой, а они скорее всего будут это делать, нам надо настроить правильно асинхронное общение между ними. Для этого используются message brokers. Мы с вами будем использовать `RabbitMQ`. Кроме него есть еще `Kafka`, `ActiveMQ`, `ZeroMQ`, `NATS` и другие. У нас в стране чаще всего используют `RabbitMQ` и `Kafka`.

## Принцип работы и цели Message Broker-ов 

`Message Broker` - это посредник между отправителем и получателем сообщения. Он принимает сообщения от отправителя и передает их получателю. Он может хранить сообщения, если получатель не готов их принять. 

Есть такие приложение где без `Message Broker` не обойтись. 

Например, в онлайн-играх, онлайн-чате, системах мониторинга и т.д. Давайте представим пример с добавлением в друзья в какой-то онлайн игре. Когда вы добавляете друга, ваше приложение отправляет сообщение на сервер, что вы хотите добавить друга. Сервер должен отправить это сообщение другу, но друг может быть оффлайн. В этом случае сервер отправит сообщение в `Message Broker`, а когда друг зайдет он получит это сообщение.


Или же если речь идет именно о большом количестве запросов с одного сервиса на другой, то `Message Broker` может помочь вам сделать это асинхронно. Тут есть уже несколько противоречий о которых нужно поговорить.

## Зачем нам `Message Broker` если есть `REST API`?

Предположим что мы в нашем проекте из `Communication Service` хотим отправить запрос в `Auth Service` для добавления в друзья. С обычным `gRPC` запросом мы будем ожидать пока это произойдет, даже если это будет быстро. Сервер для всех пользователей работает асинхронно, но клиенты ждут ответа.

С `Message Broker` мы можем отправить сообщение и продолжить работу. `Message Broker` займется отправкой сообщения и когда получатель будет готов, он получит это сообщение.

## Когда использовать `Message Broker` а когда `REST API`?

Как минимум если у вас есть что-то похожее на `Notification Service` то вам уже нужен `Message Broker`. Если у вас есть много запросов с одного сервиса на другой, то вам тоже нужен `Message Broker`.

# RabbitMQ

Есть несколько видов настройки `RabbitMQ`:
- `Direct Exchange`
- `Fanout Exchange`
- `Topic Exchange`
- `Headers Exchange`

### Direct Exchange

`Direct Exchange` - это обычный обмен сообщениями. Он принимает сообщение и отправляет его в очередь с указанным `routing key`.

### Fanout Exchange

`Fanout Exchange` - отправляет сообщение во все очереди, которые к нему подключены. `Routing key` игнорируется.

### Topic Exchange

`Topic Exchange` - отправляет сообщение в очередь, которая соответствует шаблону `routing key`. `Routing key` может содержать `*` и `#`.

### Headers Exchange

`Headers Exchange` - отправляет сообщение в очередь, которая соответствует заголовкам сообщения. 

Также в RabbitMQ есть `Dead Letter Exchange` и `Alternate Exchange`. 

Если ваш сервис упал, то `Dead Letter Exchange` дождется пока сервис поднимется и отправит сообщение, а `Alternate Exchange` отправит сообщение в другую очередь, если первая не справилась.
