## Хеширование паролей

Данная тема является одной из самых базовых направлений в области безопастности веб-приложений. Хеширование паролей - это процесс преобразования пароля в строку фиксированной длины.

Существуют разные алгоритмы хеширования, но наиболее популярными являются:

- MD5
- SHA-256
- SHA-512

Также вам нужно знать про `HMAC` (Hash-based Message Authentication Code) - это функция хеширования, которая используется для проверки целостности данных и аутентификации сообщений. Она используется в паре с секретным ключом, который известен только серверу. Давайте разберем в целом что такое хеширование, при чем тут GetHashCode() из класса Object и в чем разница между хешированием и шифрованием.

### Хеширование

У любоых данных есть свой хеш-код, который мы можем получить при пропуске их через какой-то алгоритм. При этом 90% людей путаются между хешированием и шифрованием. Дело в том, что любой шифр можно расшифровать, а хеш-код нельзя.
Хеш-код также называют One-way функцией, потому что его нельзя расшифровать.

### Шифрование

Шифрование - это процесс преобразования данных в нечитаемый формат. Шифрование используется для защиты данных от несанкционированного доступа. Шифрование использует ключ для преобразования данных в нечитаемый формат, который может быть расшифрован с использованием того же ключа.

В случае с JWT токенами мы используем и хеш-код и шифрование одновременно. Хеш-код используется для проверки целостности данных, а шифрование для защиты данных от несанкционированного доступа.

### Зачем нам все это время нужен был GetHashCode() ?

Я более чем уверен, что до сих пор вы не знали полного применения этой функции в `C#`.

`GetHashCode()` - это функция, которая возвращает хеш-код объекта. Этот хеш-код используется во многих коллекциях, таких как `Dictionary`, `HashSet`, `Hashtable` и т.д. для оптимизации поиска элементов. Он используется как для оптимизации поиска элементов, так и в самом `.NET CORE`. Вот вам пример:

```csharp

string message = "Hello, World!";

Console.WriteLine("Hello, World!");

```

В примере выше у нас есть два литерала, которые равны. Хеш-код у них тоже одинаковый соотвественно, в памяти они будут храниться как один объект. Это позволяет оптимизировать память и ускорить поиск элементов. Без этого метода вы не ссможете например хрпнить свои типы как Dictionary или HashSet ключи.

## JWT токены

`Json Web Token` - это стандарт, который определяет компактный и автономный способ безопасного передавать информацию между сторонами в виде объекта. Эта информация может быть проверена и доверена, потому что она цифровая подпись. JWT токены могут быть подписаны с использованием алгоритма HMAC (HS256), или алгоритмом RSA (RS256).

Для того чтобы понять его принцип работы вы должны быть знакомы с термином токен.

`Token` - это одноразовый ключ, который используется для аутентификации пользователя. Токены могут быть использованы для аутентификации пользователя, чтобы предоставить доступ к ресурсам. Токены могут быть использованы для аутентификации пользователя, чтобы предоставить доступ к ресурсам.

JWT токены состоят из трех частей:

- Header
- Payload
- Signature

### Header

Header содержит в себе информацию о типе шифрования и типе токена. Вот пример:

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

### Payload

В нем хранится любая информация которую я хочу послать. Например, username, role, expire date и т.д. Вот пример:

```json
{
  "sub": "1234567890", // subject
  "name": "John Doe", // name
  "iat": 1516239022 // issued at (время в секундах по UNIX)
}
```

### Signature

Signature - это подпись, которая создается на основе Header и Payload. Она создается с использованием секретного ключа. Секретный ключ должен быть известен только серверу, который будет хранится например в appsettings.json или в environment variables. Дело в том, что любой человек кто получит доступ к JWT может зайти в Debugger внутри jwt.io и посмотреть Header и Payload, но Signature он не сможет посмотреть потому что в нем используется секретный ключ.

### Как работает JWT ?

Пользователь для начала должен пройти аутентификацию, затем наш сервер генерирует
ему JTW токен и отправляет его пользователю.
Пользователь сохраняет его в localStorage или в Cookies. Важно учесть что тут
надо обязательно использовать HTTPS иначне все что мы делаем для безопастности
будет бесполезно.

После того как мы отправили пользователю JWT токен, при каждом следующем своем
запросе он отправляет его вместе с запросом.
Наш сервер подставляет свой ключ в этот токен и проверяет его подпись. Если
токен валидный, то сервер продолжает обработку запроса, если нет, то возвращает
ошибку.

## Issuer и Audience

`Issuer` - это тот кто создал токен. Например, ваш сервер.
`Audience` - это тот кто должен его использовать. Например, ваш клиент.

Мы прописываем их в настройках нашего сервера, чтобы убедиться что токен
используется только теми кто должен его использовать.
